# 11.6 四层协作：全景视图

> 前面五节分别讲了压缩管线、Auto-Compact、Session Memory、MemDir 和 autoDream。这一节把它们放在一起，展示信息如何在四个层次之间流动——在哪一层被保留、在哪一层被丢弃、各层如何互补。

## 11.6.1 时间线视图

一个完整的工作周期中，四个机制在不同时间点介入：

```
时间 ──────────────────────────────────────────────────────────────→

│ 会话开始                                                   会话结束 │

│    │  ┌─ Layer 1-3：轻量压缩 ──────────────────────────────┐  │    │
│    │  │  落盘、去重、微压缩——持续进行，几乎零成本          │  │    │
│    │  └────────────────────────────────────────────────────┘  │    │
│    │                                                          │    │
│    │         ┌─ Session Memory ────────────────────┐          │    │
│    │         │  定期更新笔记（基于消耗量）          │          │    │
│    │         └────────────────────────────────────┘          │    │
│    │                                                          │    │
│    │                   ┌─ Auto-Compact ─┐    ┌─ Auto-Compact ┐│    │

│            │  记忆文件     │                                       │
│            └──────────────┘                                        │
│                                                                    │
│ 下一次会话开始                                                     │
│    │                                                               │
│    ├── 加载 MEMORY.md 索引（MemDir）                                │
│    ├── 加载 Session Memory（如果恢复同一会话）                       │

```

## 11.6.2 信息在哪一层被保留？

不同类型的信息在不同层次有不同的"存活率"：

```
信息类型            │ 上下文窗口 │ Compact后 │ Session Memory │ MemDir
────────────────────┼────────────┼──────────┼───────────────┼────────
用户原话            │   ✅ 完整   │  ⚠️ 引用  │   ⚠️ 摘要     │  ❌
工具输出细节        │   ✅ 完整   │  ❌ 丢失  │   ⚠️ 关键的   │  ❌
当前任务状态        │   ✅ 完整   │  ✅ 保留  │   ✅ 保留     │  ❌
用户偏好/纠正       │   ✅ 完整   │  ⚠️ 可能  │   ✅ 保留     │  ✅ 持久
项目决策背景        │   ✅ 完整   │  ⚠️ 可能  │   ⚠️ 可能     │  ✅ 持久
外部系统引用        │   ⚠️ 提到时 │  ❌ 丢失  │   ❌ 通常不记  │  ✅ 持久
跨会话记忆          │   ❌ 不存在 │  ❌ 不存在 │   ❌ 会话内    │  ✅ 持久
```

关键洞察：**没有任何单一层次能保留所有信息**。四层的价值在于互补：

- **上下文窗口**：保真度最高，但容量有限
- **Compact**：延长窗口寿命，但有损
- **Session Memory**：跨 Compact 保持连续性，但会话结束就停
- **MemDir**：跨会话持久，但只存最核心的信息

## 11.6.4 设计决策与权衡

### 为什么不把所有信息都存到 MemDir？

如果每条对话都自动存入 MemDir，很快就会面临：
- **Token 爆炸**：MEMORY.md 索引需要每次注入，记忆越多 Token 越贵
- **噪音问题**：大量琐碎记忆淹没真正重要的信息
- **过时风险**：临时信息存成"持久记忆"后，需要额外的管理成本

MemDir 有意只存**跨会话仍有价值**的信息——这是一个"高门槛、低容量"的设计。

### 为什么 Session Memory 不替代 Compact？

Session Memory 是 12K tokens 的结构化笔记，不能替代 200K tokens 的完整对话——它没有足够的细节来支撑精确的代码操作。Compact 虽然有损，但保留的信息量远大于 Session Memory。

两者的关系是：**Compact 是主力，Session Memory 是保险**。

### 为什么不用 RAG（向量检索增强生成）？

当前的 MemDir 记忆通常 < 50 条。在这个规模下：
- LLM 直接判断相关性比向量检索更准确
- 不需要维护向量索引的基础设施
- 用 Sonnet 做一次筛选 < 0.5 秒，足够快

如果记忆规模增长到几百条以上，向量检索可能会变得必要。但在当前规模下，简单方案更好。

### 成本分析

```
操作                    │ Token 成本    │ 延迟       │ 频率
────────────────────────┼──────────────┼────────────┼──────────────
L1-L3 轻量压缩          │ ~0           │ < 100ms    │ 每轮
Session Memory 更新     │ ~2K output   │ 1-2s       │ 每几轮
Auto-Compact            │ ~5K output   │ 3-5s       │ 每会话 1-3 次
extractMemories         │ ~3K output   │ 2-3s       │ 每会话结束
autoDream               │ ~5K output   │ 3-5s       │ 空闲时周期性
MEMORY.md 注入          │ ~2K input    │ 0          │ 每轮（固定成本）
记忆相关性筛选          │ ~1K          │ < 0.5s     │ 有记忆时每轮
```

关键的 Fork 子 Agent 操作（Session Memory 更新、Auto-Compact、extractMemories、autoDream）都**共享父会话的 Prompt Cache**，这意味着只需要支付增量输出的 Token 费用，而不是完整对话的费用。

## 11.6.6 总结

四个机制各司其职，覆盖了从秒级到天级的记忆需求：

| 机制 | 时间尺度 | 核心职责 | 一句话 |
|------|---------|---------|--------|
| 五层压缩管线 | 秒级 | 管理窗口内的信息密度 | "桌面满了就收拾" |
| Session Memory | 分钟级 | 跨 Compact 保持连续性 | "收拾前先做笔记" |
| MemDir | 天级 | 跨会话持久化核心信息 | "重要的写进日记" |
| autoDream | 后台 | 整合和修剪持久记忆 | "睡觉时整理记忆" |

这四层的设计哲学是一致的：**用最小的信息损失覆盖最大的时间跨度**。每一层都比上一层损失更多细节，但也覆盖更长的时间——直到 MemDir 只保留最核心的、跨会话仍然有价值的信息。
